## 模块二：业务背景与商业目标
### 2.1 业务背景与核心痛点
**外部环境**：企业数字化转型加速，AI大模型技术日趋成熟，为企业内部知识管理和办公效率提升带来全新机遇。公司明确提出"数字员工人机协同"战略，推动AI赋能内部业务流程。
**内部瓶颈（具体、可验证）**：
| 痛点编号 | 痛点描述 | 影响范围 | 可验证方式 | 对应功能 |
|---------|---------|---------|----------|---------|
| P1 | **信息碎片化，检索效率低**：飞书中的文档、会议纪要、聊天记录分散在各处，员工平均每天花费1.5小时翻找，缺乏统一归档和语义检索机制 | 全体员工 | 抽样调查问卷 | ETL同步、工作配方同步、机器人归档、AI问答（RAG）、全局搜索 |
| P2 | **数据质量差，加工全靠人工**：各业务线多维表格格式各异（日期、数字、命名不统一），存在重复行、空行、冗余列等问题；同时不同行业关注的关键信息差异大（如能源关注装机容量，城建关注用地面积），从文档中提取关键字段完全依赖人工阅读，效率低且易遗漏 | 各业务线员工、数据分析人员 | 数据清洗返工次数 + 关键字段遗漏率统计 | 提取规则引擎（4种行业模板）、清洗规则引擎（5种规则化+3种AI智能清洗）、ETL智能字段映射 |
| P3 | **知识断层，协作信息难追溯**：跨部门项目信息传递依赖口头或重复发送，关键信息遗漏率约30%；会议中产生的行动项容易被遗忘，缺乏从纪要到待办的自动化跟踪链路；人与项目的关联关系隐藏在文字中难以梳理 | 项目协作人员、项目团队 | 项目复盘数据 + 待办完成率对比 | 知识图谱、人物洞察分析、自动待办提取、日程提醒 |
| P4 | **决策支撑慢，报告依赖人工**：管理层需要人工汇总周报/月报，每份报告耗时2-3小时；缺乏对团队数据资产的全局视图和趋势洞察 | 管理层 | 报告编制时间统计 | 报告自动生成、数据洞察看板 |
| P5 | **权限管控粗放，数据安全存隐患**：缺乏按部门/个人灵活配置数据可见范围的机制，AI问答也无法感知权限边界 | IT管理/管理层 | 权限申诉工单数 | 部门与权限管理、RLS行级安全、AI权限感知 |
### 2.2 产品目标与预期收益
**定性目标**：构建企业级智能数据中枢——"流光·办公副驾"，自动归集飞书生态中的文档、会议、聊天等多源数据，通过AI智能处理将其转化为可检索、可问答、可分析的数据资产，从根本上解决企业内部信息碎片化和知识孤岛问题。
**定量指标（北极星指标）**：
| 指标名称 | 当前基线 | 目标值 | 衡量方式 |
|---------|---------|-------|----------|
| 信息检索效率 | 平均15分钟/次 | 6分钟/次（提升60%） | 用户操作日志分析 |
| 报告生成时间 | 人工2-3小时/份 | AI自动5分钟/份 | 报告生成时间埋点 |
| 用户周活跃率（WAU） | 0（新产品） | 70% | 登录埋点统计 |
| 数据归档覆盖率 | 手动零散归档 | 90%飞书核心数据源 | 数据源同步状态统计 |
| 待办遗漏率 | 约40%（人工记录） | 降至10%以下 | 待办提取vs实际行动项对比 |
### 2.3 竞品分析与差异化优势
| 对比维度 | 飞书原生搜索 | Notion AI | 流光·办公副驾 |
|---------|------------|-----------|-------------|
| 数据范围 | 仅飞书内文档搜索 | 仅Notion内笔记 | 飞书文档+会议+聊天+多维表格+上传文件，跨源聚合 |
| 搜索方式 | 关键词精确匹配 | 关键词+AI问答 | 混合检索（向量语义+BM25关键词+RRF融合），准确率更高 |
| 知识图谱 | 无 | 无 | 自动构建人物-项目-组织关系图谱，可视化展示 |
| 会议待办提取 | 需手动记录 | 无 | AI自动从会议纪要提取待办+负责人+截止日期 |
| 数据权限 | 平台级粗粒度 | 工作区级 | 行级权限（RLS），AI回答也经权限过滤 |
| 报告生成 | 无 | 简单摘要 | 指定范围自动汇总周报/月报，支持发布到飞书 |
| 部署方式 | SaaS（数据在飞书） | SaaS（数据在境外） | 私有化部署，数据不出企业环境 |
| 定制性 | 不可定制 | 有限 | 4种行业提取模板+8种清洗选项（含3种AI智能清洗）+自定义ETL配置 |
**核心差异化**：流光不是"另一个搜索工具"，而是企业级数据中枢——将散落在飞书各处的信息**自动归集、AI加工、权限隔离、智能问答**，形成闭环。飞书原生能力无法覆盖"跨数据源语义检索"和"知识图谱自动构建"两个核心场景。
### 2.4 ROI量化估算
基于30人核心团队试点规模的保守估算：
| 效率提升项 | 当前耗时 | 使用流光后 | 节省时间/人/天 | 团队年化节省 |
|-----------|---------|-----------|--------------|------------|
| 信息检索 | 1.5h/天 | 0.5h/天 | 1小时 | 30人 × 1h × 250天 = **7,500工时/年** |
| 报告编制 | 2h/份（管理层周报） | 5min/份 | — | 10人 × 2h × 50周 = **1,000工时/年** |
| 会议待办跟进 | 15min/次手动记录 | 自动提取 | — | 30人 × 3次/周 × 10min × 50周 = **750工时/年** |
| **合计** | — | — | — | **≈9,250工时/年** |
按人均时薪100元估算，年化节省约**92.5万元**人力成本。随着接入数据源增多和覆盖人数扩大，ROI将持续增长。
### 2.5 MVP与商业假设
**核心商业假设**：
- 飞书生态中的散落数据（文档、会议纪要、聊天记录）一旦被结构化归集并赋予AI检索能力，将显著提升员工信息获取效率和管理层决策速度
- 企业用户愿意使用AI问答替代手动翻找历史信息的方式
**目标用户群体**：公司全体员工（首批覆盖30-50人核心团队）
**商业模式**：内部赋能工具，通过降低信息检索时间成本和报告生成成本，实现人效提升。未来具备SaaS产品化潜力。

### 3.2 核心用户故事
**US-01 智能问答**
> 作为【普通员工】，我希望【在AI助手中用自然语言提问，就能从文档、会议纪要、聊天记录中找到答案】，以便【快速了解项目进展而不需要逐个翻找历史消息】。
**US-02 报告自动生成**
> 作为【管理员】，我希望【指定时间范围和数据范围后，一键生成部门周报或月报】，以便【将每份报告的编制时间从2小时缩短到5分钟】。
**US-03 机器人归档**
> 作为【普通员工】，我希望【直接给流光机器人发消息或转发重要内容，即可自动归档到平台】，以便【事后可以轻松搜索和追溯关键决策信息】。
**US-04 知识图谱洞察**
> 作为【管理员】，我希望【查看自动生成的人物关系图谱和沟通画像】，以便【了解团队协作模式、发现沟通瓶颈和核心节点人物】。
**US-05 自动待办提取**
> 作为【普通员工】，我希望【系统自动从我参与的会议纪要中识别并生成待办事项，同步到飞书任务】，以便【不遗漏任何行动项，提高执行力】。
**US-06 权限安全管理**
> 作为【管理员】，我希望【按部门灵活配置数据可见范围，确保员工只能看到授权数据】，以便【满足企业数据安全合规要求】。
### 3.3 Aha Moment（啊哈时刻）
**核心高光时刻**：用户首次在AI助手中用自然语言提问（如"上周XX项目有什么进展？"），系统从多个数据源（文档 + 会议纪要 + 聊天记录）综合检索后给出答案，并且每条信息都标注了来源出处——用户点击来源即可跳转到原始内容。
**这一刻的价值感知**：
- "原来AI真的能理解我的问题"
- "不用再花15分钟翻聊天记录了"
- "有来源引用，答案可信"
> 提示UI/UX：在此环节需重点打磨——来源引用的可点击性、回答的流式输出体验、首次使用的引导提示。

### 4.3 状态机图（State Machine）
#### 状态机一：ETL同步状态
```
                    ┌─────────┐
                    │  待同步   │ ← 初始状态（新建数据源）
                    └────┬────┘
                         │ 触发同步（定时/手动）
                         ▼
                    ┌─────────┐
                    │  同步中   │
                    └──┬───┬──┘
           同步成功 │       │ 同步失败
                   ▼       ▼
             ┌────────┐ ┌────────┐
             │ 已完成  │ │ 失败   │
             └───┬────┘ └───┬────┘
                 │          │ 可重试
                 │          ▼
                 │     ┌─────────┐
                 └────→│  待同步  │ ← 等待下次同步周期
                       └─────────┘
```
> 无永久终点，为循环状态机（持续增量同步）
#### 状态机二：待办事项状态
```
    ┌─────────┐
    │  待审核   │ ← 初始状态（AI自动提取生成）
    └──┬───┬──┘
       │   │ 用户忽略
       │   ▼
       │ ┌─────────┐
       │ │  已忽略   │ ← 终点状态
       │ └─────────┘
       │ 用户确认
       ▼
    ┌─────────┐
    │  进行中   │
    └────┬────┘
         │ 标记完成
         ▼
    ┌─────────┐
    │  已完成   │ ← 终点状态
    └─────────┘
```
#### 状态机三：报告状态
```
    ┌─────────┐
    │  生成中   │ ← 初始状态（用户触发报告生成）
    └──┬───┬──┘
       │   │ 生成失败
       │   ▼
       │ ┌─────────┐
       │ │  失败    │ → 可重新触发 → 回到「生成中」
       │ └─────────┘
       │ 生成成功
       ▼
    ┌─────────┐
    │  已生成   │
    └────┬────┘
         │ 发布到飞书
         ▼
    ┌─────────┐
    │  已发布   │ ← 终点状态
    └─────────┘
```

#### F03-A 提取规则引擎（ExtractionRule）

**适用对象**：文档（Document）、通信记录（Communication）
**界面描述**：提取规则编辑器为模态对话框，包含四个区域——规则名称输入、行业板块选择（4个彩色按钮）、字段编辑列表（可增删）、额外提取提示（文本框）。
**交互逻辑**：
1. 新建规则：点击"新建提取规则" → 弹出编辑器
2. 选择板块：点击行业板块按钮（可多选） → 自动追加该板块推荐字段到字段列表
3. 编辑字段：在字段列表中添加/编辑/删除字段，每个字段需填写名称和描述
4. 填写提示：可选填额外提取提示（如"优先从标题和关键段落中提取"），辅助LLM理解
5. 保存并激活：保存规则 → 在规则卡片上点击激活 → 后续导入数据时自动应用
6. 查看提取数据：点击规则卡片 → 查看该规则下所有已提取的数据汇总表（支持搜索和分页）
7. 导出数据：点击"导出Excel" → 将提取结果导出为结构化表格
**字段定义（提取规则）**：
| 字段 | 类型 | 必填 | 长度/范围 | 说明 |
|------|------|------|----------|------|

| is_active | 布尔 | 是 | — | 规则激活状态（默认true） |
| owner_id | 字符串 | 是 | ≤64字符 | 规则所有者（RLS隔离） |

| 字段 | 类型 | 必填 | 说明 |

| key | 字符串 | 是 | 字段编码（如project_name） |

**4种行业板块模板**：
| 板块 | 推荐字段 |
|------|---------|
| 通用（Common） | 项目名称、关键决策、行动项、负责人 |

**执行原理**：
```
数据导入时（ETL Loader阶段）：
  ├─ 检查数据源是否关联 extraction_rule_id
  ├─ 加载提取规则 → 读取 fields 字段定义
  ├─ 构建LLM Prompt：
  │   "请从以下内容中，按照指定的字段定义提取关键信息"
  │   + 字段定义（label + description）
  │   + prompt_hint（额外提示）
  │   + 文档标题和正文
  ├─ 调用LLM → 返回JSON格式的提取结果
  └─ 写入 Document.key_info / Communication.key_info（JSONB字段）
```
**业务规则**：
- 标题是最重要的信息来源，LLM优先从标题提取核心信息
- 提取结果存储为key_info（JSONB），支持任意结构的键值对
- 同一规则下的所有提取数据可在`/data`视图中统一查看和搜索
- 规则按owner_id隔离，用户只能看到和操作自己创建的规则
#### F03-B 清洗规则引擎（CleaningRule）
**功能定位**：清洗规则是流光对**结构化表格数据**的自动化质量治理能力——用户组合选择8种清洗选项（5种规则化+2种LLM智能+1种阈值配置），系统自动执行数据清洗，消除重复、空值、格式不一致等数据质量问题。
**适用对象**：结构化表格（StructuredTable）
**界面描述**：清洗规则编辑器为模态对话框，包含三个区域——规则名称输入、8个清洗选项开关（含阈值滑块）、字段处理描述（文本框）。
**交互逻辑**：
1. 新建规则：点击"新建清洗规则" → 弹出编辑器
2. 配置选项：通过开关启用/关闭各清洗选项（默认全部启用）
3. 设置阈值：拖动"空值阈值"滑块（0%-100%），控制空行删除的敏感度
4. 填写字段描述：可选填字段处理描述（如"合并「省」「市」「区」为「地区」"），触发LLM智能字段操作
5. 保存并激活：保存规则 → 在规则卡片上点击激活
6. 手动触发清洗：在结构化表格详情页，点击"应用清洗规则"手动执行
7. 查看清洗统计：执行后展示清洗前后对比（行数变化、字段变化、LLM操作记录）
**字段定义（清洗规则）**：
| 字段 | 类型 | 必填 | 长度/范围 | 说明 |
|------|------|------|----------|------|
| name | 字符串 | 是 | ≤128字符 | 规则名称（如"标准表格清洗"） |
| options | JSON对象 | 是 | 8个选项 | 清洗选项开关组合（见下方子表） |
| field_hint | 长文本 | 否 | ≤2000字符 | 字段处理描述，引导LLM执行智能字段操作 |
| is_active | 布尔 | 是 | — | 规则激活状态（默认true） |
| owner_id | 字符串 | 是 | ≤64字符 | 规则所有者（RLS隔离） |
**8种清洗选项详解**：
| 选项 | 默认 | 类型 | 说明 |
|------|------|------|------|
| dedup | 开启 | 规则化 | 基于MD5哈希去除完全重复的行 |
| drop_empty_rows | 开启 | 规则化 | 按阈值移除空值比例过高的行 |
| empty_threshold | 50% | 阈值 | 空值比例超过此值的行将被删除（0%-100%可调） |
| normalize_dates | 开启 | 规则化 | 统一日期格式为YYYY-MM-DD（支持识别多种输入格式） |
| normalize_numbers | 开启 | 规则化 | 去除千分位逗号、单位文字（万元/亿/%），仅保留数值 |
| trim_whitespace | 开启 | 规则化 | 去除所有字符串字段的首尾空格和多余空白 |
| llm_field_merge | 开启 | LLM智能 | AI自动识别并合并含义相同的列（如"联系电话"和"手机号"） |
| llm_field_clean | 开启 | LLM智能 | AI修正错别字、统一表述风格（如"已完成""完成了""done"统一为"已完成"） |
**执行原理（分层清洗）**：
```
清洗规则执行时（结构化数据导入后自动触发）：

第一层：规则化清洗（确定性，不依赖LLM，快速可靠）
  ├─ trim_whitespace → 去除所有字段首尾空格
  ├─ normalize_dates → YYYY/MM/DD、YYYY年M月D日 → YYYY-MM-DD
  ├─ normalize_numbers → "1,234.56万元" → "12345600"
  ├─ drop_empty_rows → 空值比例 > empty_threshold 的行删除
  └─ dedup → 对每行JSON计算MD5，去除完全重复行

第二层：LLM智能清洗（仅在启用field_hint或LLM选项时触发）
  ├─ 分析field_hint → LLM返回操作指令（JSON格式）：
  │   {
  │     "drop_fields": ["要删除的冗余字段"],
  │     "rename_fields": {"旧字段名": "新字段名"},
  │     "merge_fields": [{"sources": ["省","市","区"], "target": "地区"}]
  │   }
  └─ 执行LLM操作 → 删除/重命名/合并字段

清洗后返回统计：
  清洗前行数 → 清洗后行数 | 清洗前列数 → 清洗后列数 | 执行的LLM操作列表
```
**业务规则**：
- 规则化清洗和LLM智能清洗分层执行，规则化优先保障基础数据质量
- 清洗规则可复用——同一规则可关联多个数据源
- 后续同步时自动重新应用清洗规则，保持数据质量持续治理
- 规则按owner_id隔离，用户只能看到和操作自己创建的规则
**业务规则（F03整体）**：
- 同步策略：增量同步，基于上次同步时间戳过滤
- Schema映射缓存：相同表结构的AI映射结果使用MD5缓存，避免重复LLM调用
- 提取规则应用于文档/通信类数据，清洗规则应用于结构化表格类数据
- 数据源首次配置时可同时关联提取规则和清洗规则，后续同步自动应用
#### F04 飞书机器人归档
**工作方式**：用户直接给流光机器人发消息（私聊）或转发内容给机器人，机器人智能识别内容类型并自动归档到平台。
**交互逻辑**：
1. 用户直接给流光机器人发消息或转发内容（私聊方式）
2. 飞书通过事件订阅（im.message.receive_v1）推送到后端
3. 后端验证签名 → 解析消息内容和附件
4. 机器人即时回复"收到！正在智能分析内容并自动入库，请稍候..."
5. 后台异步执行：LLM智能分类 → ETL流水线处理 → 归档入库
6. 处理完成后发送交互卡片，展示归档结果，支持用户调整分类
**业务规则**：
- 支持归档的内容类型：文本消息、图片、文件（PDF/Word/Excel）
- 机器人通过LLM智能识别内容应归入哪个资产类型（文档/通信/结构化表）
- 归档后自动生成向量嵌入，可通过AI问答检索
- 归档数据的owner_id为发送消息的用户
- 用户需先登录流光平台，机器人通过飞书open_id验证身份
- 支持通过卡片交互调整资产类型、提取规则、清洗规则
#### F05 AI智能问答（核心功能）
**界面描述**：类ChatGPT对话界面。左侧为历史对话列表（可新建、删除、重命名），右侧为当前对话区。输入框支持文字输入和文件上传。底部有多个标签页切换：对话、待办、图谱、日历。
**交互逻辑**：
1. 输入问题 → 发送请求
2. 系统执行混合检索（向量搜索 + BM25关键词搜索）
3. RRF融合排序 → 取Top N最相关内容
4. 检索结果经权限过滤（RLS） → 构建Prompt发送LLM
5. 回答通过SSE流式返回 → 前端逐字渲染
6. 回答底部展示来源引用 → 点击可跳转原始内容页面
**字段与规则**：
| 字段 | 类型 | 限制 | 说明 |
|------|------|------|------|
| 用户问题 | 文本 | ≤2000字符 | 支持中英文混合 |
| 上传附件 | 文件 | ≤50MB，PDF/Word/TXT/CSV/图片 | 可基于附件内容提问 |
| 回答 | 流式文本 | — | SSE逐字输出，支持Markdown格式 |
| 来源引用 | 列表 | Top 5条 | 包含来源类型、标题、内容ID |
| 对话上下文 | 历史消息 | 最近10轮 | 多轮对话记忆 |
**排序规则**：检索结果按RRF（Reciprocal Rank Fusion）分数降序排列，取Top 5作为LLM上下文。
**计算公式**：`RRF_score = Σ(1 / (k + rank_i))`，其中k=60（常数），rank_i为该结果在各检索通道中的排名。

| target_type | 枚举 | department / user |
| target_id | 数字 | 目标部门ID或用户ID |
| permission_level | 枚举 | read（只读查看） |
| created_by | 数字 | 配置人（管理员） |
**业务规则**：
- 部门结构从飞书通讯录API同步，不支持手动创建部门
- 超级管理员（SUPER_ADMIN_OPEN_ID）角色不可被修改或降级
- 共享规则变更即时生效，影响RLS查询过滤
- 非管理员访问此页面返回403
### 5.3 异常与边界处理
#### 空状态（Empty）
| 场景 | 处理方式 |
|------|----------|
| 首次登录无任何数据 | 展示欢迎引导页，提示"点击添加数据源开始同步"，附操作示意图 |
| 搜索/筛选无结果 | 展示空状态插图+文案"未找到匹配内容，试试其他关键词" |
| AI问答无相关资料 | AI回答"暂未找到相关信息，建议同步更多数据源或尝试不同提问方式" |
| 知识图谱无实体 | 展示提示"数据量不足，系统将在数据积累后自动构建图谱" |
| 待办列表为空 | 展示"暂无待办事项"，提示可手动创建或等待系统自动提取 |
#### 极限与并发（Extreme）
| 场景 | 处理方式 |
|------|----------|
| 弱网/断网 | 前端显示网络异常提示toast，自动重试3次（指数退避：1s/2s/4s） |
| ETL并发同步 | 同一数据源加分布式锁，防止重复同步；不同数据源可并行 |
| 按钮防重复提交 | 点击后按钮立即禁用+显示loading状态，请求返回后恢复 |
| 文本超长 | 文档内容超过10万字时分块处理向量化；列表中文本超200字截断+展开按钮 |
| 大量数据加载 | 列表默认分页（每页20条）；超长列表使用虚拟滚动 |
| LLM响应超时 | 30秒超时限制，超时后提示"AI正忙，请稍后重试" |
| 文件上传超限 | 超过50MB提示"文件过大，请压缩后重试"，不发起请求 |
#### AI可靠性保障
| 场景 | 处理方式 |
|------|----------|
| LLM幻觉（编造不存在的信息） | RAG强制基于检索结果生成，Prompt中明确要求"仅基于以下资料回答，无相关资料时回复不知道"；每条回答附带来源引用，用户可自行核实 |
| 检索无相关结果 | AI回答"暂未找到相关信息"而非编造内容，并建议用户调整提问或补充数据源 |
| LLM返回格式异常 | 后端解析失败时返回兜底文案"AI处理异常，请重新提问"，前端正常展示 |
| ETL同步连续失败 | 连续3次同步失败 → 自动触发飞书Webhook告警（通知管理员群） → 暂停该数据源同步，等待人工排查 |
| 飞书API限流/不可用 | 指数退避重试（1s/2s/4s），3次失败后标记同步状态为"失败"并记录错误详情 |
| LLM API额度耗尽 | 检测429状态码 → 降级为仅关键词搜索（跳过LLM生成） → 提示"AI服务暂时不可用，已切换为关键词搜索模式" |
#### 权限与合规边界
| 场景 | 处理方式 |
|------|----------|
| 未登录访问任何页面 | 自动重定向到 /login 登录页 |
| JWT Token过期（401） | 清除本地token → 自动跳转登录页，不打断用户操作提示 |
| 无权限访问数据（403） | 弹出提示"您无权访问此内容，如需权限请联系管理员" |
| 非企业飞书用户 | 飞书SSO企业域限制，非企业成员无法完成授权 |
| 特殊字符防注入 | 所有用户输入经Pydantic严格校验；SQL全部使用参数化查询（SQLAlchemy ORM） |
| 管理员模式切换 | 管理员需手动开启"管理员模式"（X-Admin-Mode头），默认以普通视角查看 |
## 模块六：非功能性需求（NFR）与数据埋点
### 6.1 性能与兼容性要求
| 指标 | 要求 |
|------|------|

| API接口响应 | 常规接口 ≤ 500ms，复杂查询 ≤ 2s |
| SSE首字延迟 | AI问答首字输出 ≤ 3秒 |
| 并发支持 | 50并发用户（首批团队规模） |
| 浏览器兼容 | Chrome、Edge、Safari 最新两个版本 |
| 最低分辨率 | 1280×720，推荐1920×1080 |
| 文件上传限制 | 单文件最大50MB |
### 6.2 安全与合规要求
| 安全项 | 实施方案 |
|--------|----------|
| 身份认证 | 飞书OAuth 2.0 SSO，企业域限制，仅企业成员可访问 |
| 会话管理 | JWT（HS256算法），24小时自动过期 |
| 数据权限 | 行级安全（RLS），每条数据绑定owner_id，自动过滤 |
| 传输加密 | HTTPS（TLS 1.2+），全站强制 |
| 密钥管理 | 敏感配置通过环境变量注入，不入代码仓库 |
| AI权限感知 | RAG检索自动过滤当前用户无权限的数据，确保AI不泄露未授权信息 |
| 日志安全 | 日志中不记录用户完整JWT token、密码等敏感信息 |
| 数据归属 | 所有数据不出企业环境，存储在自建PostgreSQL数据库中 |

### A. 不做的范围（Out of Scope）
- 企业微信/钉钉等非飞书平台数据源接入
- 移动端原生App（当前仅Web端）
- 多租户SaaS化部署架构
- 离线/私有化大模型部署
- OA审批流程引擎
- 实时音视频会议功能
- 与第三方ERP/CRM系统深度集成
### B. 名词解释
| 名词 | 全称 | 解释 |
|------|------|------|
| ETL | Extract-Transform-Load | 数据抽取-转换-加载，数据处理的标准流程 |
| RAG | Retrieval-Augmented Generation | 检索增强生成，先检索相关资料再让AI生成回答 |
| SSO | Single Sign-On | 单点登录，用飞书账号一键登录 |
| JWT | JSON Web Token | 身份验证令牌，证明"你是谁" |
| RLS | Row-Level Security | 行级安全，每条数据都有权限控制 |
| SSE | Server-Sent Events | 服务端推送事件，实现流式输出效果 |
| RRF | Reciprocal Rank Fusion | 互惠排名融合，合并多路搜索结果的算法 |
| pgvector | PostgreSQL Vector Extension | PostgreSQL的向量检索扩展，支持语义搜索 |
| LLM | Large Language Model | 大语言模型，如GLM、DeepSeek等 |
| KG | Knowledge Graph | 知识图谱，实体和关系的网络 |
| RBAC | Role-Based Access Control | 基于角色的权限控制 |
| MECE | Mutually Exclusive, Collectively Exhaustive | 相互独立、完全穷尽，需求分析原则 |
