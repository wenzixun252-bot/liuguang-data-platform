\# 飞书企业级 ETL 与“流光”智能数据资产平台全栈架构 (PRD V3 终极版)



\## 1. 项目定位与业务目标

本项目旨在开发一个企业级的全栈 Web 应用系统。核心业务闭环如下：

1\. \*\*隐性知识资产化：\*\* 通过后台静默运行的 ETL 引擎，将员工散落在飞书多维表格中的会话记录、会议纪要、归档文档等数据，自动抓取、清洗并沉淀为标准化的企业数据资产。

2\. \*\*安全隔离访问：\*\* 通过飞书单点登录 (SSO) 实现员工身份绑定，在应用层构建严格的行级数据权限隔离（普通员工仅见个人数据，高管/管理员具备全局视角）。

3\. \*\*“流光”智能应用：\*\* 基于结构化入库的数据资产，构建内嵌的“流光”办公助手智能体，提供带有权限感知的 RAG（检索增强生成）问答服务与专属数据看板。



\## 2. 技术栈选型规范

请严格按照以下技术栈进行系统架构设计与代码输出：

\* \*\*后端框架：\*\* FastAPI (Python 3.10+，充分利用其异步特性和对 LLM 流式输出的友好支持)。

\* \*\*关系型与向量数据库：\*\* PostgreSQL (必须启用 `pgvector` 扩展插件)。

\* \*\*数据库交互 (ORM)：\*\* 异步 SQLAlchemy。

\* \*\*后台任务调度：\*\* APScheduler 或 Celery + Redis (用于管理定时抓取的 ETL 任务)。

\* \*\*大模型交互：\*\* \* ETL 清洗层：接入 DeepSeek 或 Qwen 等高性价比大模型 API（用于 Schema 语义映射）。

&nbsp; \* Agent 应用层：预留标准 OpenAI 格式接口，用于驱动“流光”智能体。

\* \*\*飞书开放平台 SDK：\*\* 使用官方 `larksuite-oapi` 处理 API 请求与 Webhook 告警。



\## 3. 核心业务模块与逻辑定义



\### 3.1 统一身份认证与权限控制 (Auth \& RBAC)

\* \*\*飞书免登 (SSO)：\*\* 前端引导用户进行飞书 OAuth 2.0 授权，后端通过临时 `code` 换取 `user\_access\_token`，进而获取并绑定用户的 `feishu\_open\_id`。

\* \*\*应用内鉴权：\*\* 后端签发 JWT Token，并在数据库 `Users` 表中维护用户角色（普通员工 `employee` / 高管 `executive`）。所有业务接口必须依赖验证该 Token。



\### 3.2 自动化 ETL 数据抓取引擎 (Extract)

\* \*\*注册中心寻址：\*\* 定时读取全局《ETL 抓取注册中心》（飞书多维表格），获取目标 `app\_token` 和表类型。

\* \*\*增量拉取策略：\*\* 使用企业自建应用凭证 (`tenant\_access\_token`)，每次仅抓取目标表中 `update\_time` 大于上次同步时间的增量记录，严格控制飞书 API 调用频次。



\### 3.3 LLM 动态清洗与预警拦截 (Transform)

\* \*\*智能字段映射：\*\* 每次抓取前拉取源表 Schema。将源 Schema 与目标标准 Schema 构建 Prompt，调用性价比大模型进行语义对齐（如将“核心提要”映射为 `core\_content`）。

\* \*\*映射缓存机制：\*\* 必须在本地或 Redis 中对 Schema 的 MD5 值和映射结果进行缓存，严禁对同一表结构重复调用 LLM。

\* \*\*异构数据处理：\*\* 将源表中无法直接对齐的 AI 标签或个性化字段，统一打包存入 PostgreSQL 的 `asset\_tags` (JSONB) 字段中。

\* \*\*异常告警：\*\* 遇关键字段缺失或 LLM 映射彻底失败时，在内存中丢弃该条数据记录日志，并通过飞书 Webhook 向管理员发送包含“错误链接”和“失败原因”的告警卡片。



\### 3.4 资产沉淀与数据库写入 (Load)

\* \*\*统一表结构标准：\*\* 无论是会话、纪要还是文档资产表，必须包含以下核心字段：

&nbsp; \* `feishu\_record\_id` (VARCHAR, PRIMARY KEY) - 确保全局唯一。

&nbsp; \* `owner\_id` (VARCHAR) - 记录数据归属人的飞书 `open\_id`，作为后续权限隔离的锚点。

&nbsp; \* `content\_text` (TEXT) - 原始明文数据。

&nbsp; \* `content\_vector` (VECTOR) - 调用 Embedding 模型生成的文本向量。

&nbsp; \* `asset\_tags` (JSONB) - 灵活扩展标签。

\* \*\*入库策略：\*\* 必须采用 `INSERT ... ON CONFLICT (feishu\_record\_id) DO UPDATE ...` 的 Upsert 逻辑，避免重复数据产生。



\### 3.5 专属看板与“流光”助手 (App Layer \& RAG)

\* \*\*应用层权限隔离 (RLS)：\*\* 所有针对普通员工的数据查询接口，必须在 ORM 层面强制附加 `WHERE owner\_id = '当前登录用户的 feishu\_open\_id'` 过滤条件。

\* \*\*权限感知的混合检索：\*\* “流光”智能体在执行 RAG 时，需采用混合检索机制（Vector 向量相似度 + BM25 关键词检索）。在检索底层数据库时，必须将当前用户的 `owner\_id` 权限作为硬性过滤前置条件，确保 LLM 生成的回答不发生数据越权。
