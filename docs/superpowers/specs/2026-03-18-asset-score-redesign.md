# 数据评分体系重新设计 — 设计文档

## 概述

重新设计数据评分体系，从原有 7 维度简单平均改为 5 维度加权评分，聚焦数据质量。每个维度包含子指标，支持展开查看评分标准和用户当前完成情况。评分将用于奖金发放参考，因此需要高区分度（大部分人 40-70 分）。

## 核心变更

| 项目 | 旧方案 | 新方案 |
|------|--------|--------|
| 维度数 | 7 | 5 |
| 总分计算 | 简单平均 | 加权求和 |
| 参评数据 | 全部 3 张表 | 文档 + 表格 + 会议录音（排除会话） |
| 子指标 | 无 | 每个维度 2-3 个子指标 |
| 评分标准 | 不可见 | 可展开查看梯度和用户位置 |
| 区分度 | 低（多维度 0 或 100） | 高（40-70 为主要分布区间） |

## 参评数据范围

- **Document**：全部参评
- **StructuredTable**：全部参评
- **Communication**：仅 `comm_type` 为会议/录音类型参评，会话记录排除
- **被归档次数**：仅统计 Document + StructuredTable

## 5 个评分维度

### 1. 内容质量（权重 30%）

衡量数据内容本身的质量水平。

| 子指标 | 子权重 | 数据来源 | 计算方式 |
|--------|--------|---------|---------|
| ETL 质量均分 | 40% | 参评内容的 `quality_score` | 平均值阶梯映射（见下方） |
| 高质量内容占比 | 35% | `quality_score >= 0.7` 的记录数 | 占总参评内容百分比阶梯映射 |
| 字段完整率 | 25% | 标题字段 + `content_text` 均非空 | 占比 × 100 线性映射 |

> **注意**：Document 和 Communication 的标题字段为 `title`，StructuredTable 的标题字段为 `name`。

**ETL 质量均分梯度：**

| 均值区间 | 得分（满分 40，= 100 × 子权重 0.4） |
|---------|--------------------------------------|
| >= 0.85 | 36-40 |
| 0.7 - 0.85 | 28-36 |
| 0.5 - 0.7 | 20-28 |
| 0.3 - 0.5 | 12-20 |
| < 0.3 | 0-12 |

采用区间内线性插值，而非硬跳档。

**高质量内容占比梯度：**

| 占比区间 | 得分（满分 35，= 100 × 子权重 0.35） |
|---------|---------------------------------------|
| >= 70% | 30-35 |
| 50-70% | 22-30 |
| 30-50% | 15-22 |
| 10-30% | 8-15 |
| < 10% | 0-8 |

> **子指标满分规则**：每个子指标的满分 = 100 × 子权重。例如子权重 0.4 的指标满分 40 分，子权重 0.25 的满分 25 分。维度总分 = 各子指标得分之和（0-100）。

**典型得分**：35-55 分。quality_score 由 ETL 管线的 postprocessor 自动计算（内容长度 40% + 字段完整度 35% + 无乱码 25%），用户无法直接刷分。

### 2. 数据完备度（权重 20%）

衡量数据资产是否全面、均衡。

| 子指标 | 子权重 | 数据来源 | 计算方式 |
|--------|--------|---------|---------|
| 数据量 | 50% | 参评内容总数 | `log(count+1) / log(1001) × 100`，需约 1000 条满分 |
| 类型覆盖 | 25% | 3 种内容类型各是否有数据 | 1 种→40, 2 种→70, 3 种→100 |
| 数据源数量 | 25% | ETLDataSource + CloudFolderSource 配置数 | `log(count+1) / log(11) × 100`，10 个满分 |

**典型得分**：35-60 分。需要 1000 条数据（旧方案 500 条）才能拿满数据量子项，且需要文档、表格、会议三种类型都有数据。

### 3. 标签规范度（权重 20%）

衡量数据标签治理水平。

| 子指标 | 子权重 | 数据来源 | 计算方式 |
|--------|--------|---------|---------|
| 标签覆盖率 | 40% | ContentTag 中参评内容的覆盖数 | `tagged_count / scored_count × 100` |
| 标签多样性 | 35% | 使用的不同标签种类数 | `min(distinct_tags, 10) / 10 × 100` |
| 标签深度 | 25% | 被标签内容的平均标签数 | `min(avg_tags_per_item, 3) / 3 × 100` |

**典型得分**：25-55 分。全部内容打同一个标签只能拿到覆盖率分，多样性和深度分很低。需要使用 10+ 种标签、每条内容平均 3 个标签才能满分。

### 4. 数据时效性（权重 15%）

衡量数据是否保持更新，防止「导入一次就躺平」。

| 子指标 | 子权重 | 数据来源 | 计算方式 |
|--------|--------|---------|---------|
| 近期更新率 | 60% | 近 30 天 `created_at` 的参评内容 | `recent_count / total_count × 100` |
| 更新规律性 | 40% | 近 30 天有新数据入库的天数 | `days_with_data / 30 × 100` |

**典型得分**：20-60 分。需要持续稳定地同步/上传数据。一次性批量导入只能拿更新率分，规律性分很低。

### 5. 数据影响力（权重 15%）— 仅文档 + 表格

衡量用户原创数据被他人认可的程度。

| 子指标 | 子权重 | 数据来源 | 计算方式 |
|--------|--------|---------|---------|
| 被归档总次数 | 50% | 他人归档我原创内容的记录数 | `log(count+1) / log(51) × 100`，50 次满分 |
| 独立引用人数 | 30% | 归档我内容的不同用户数 | `log(users+1) / log(21) × 100`，20 人满分 |
| 被引内容覆盖率 | 20% | 被至少 1 人归档的原创内容占比 | `referenced_count / my_original_count × 100` |

**数据来源逻辑**：通过 `extra_fields._original_owner.id` 识别原创者，统计 `owner_id != 原创者` 的记录为"被他人归档"。

**典型得分**：0-40 分。这是最难拿高分的维度，需要产出真正有价值的内容让同事也归档。

## 总分计算

```
total_score = 内容质量分 × 0.30
            + 数据完备度分 × 0.20
            + 标签规范度分 × 0.20
            + 数据时效性分 × 0.15
            + 数据影响力分 × 0.15
```

## 等级划分

| 分数区间 | 等级 | 颜色方案 |
|---------|------|---------|
| >= 90 | 卓越 | emerald (绿色渐变) |
| >= 70 | 优秀 | indigo-purple (品牌渐变) |
| >= 50 | 良好 | amber (琥珀色) |
| < 50 | 待提升 | red (红色) |

## 预期分数分布

| 用户类型 | 预期总分 | 说明 |
|---------|---------|------|
| 刚开始使用 | 10-25 | 少量数据，无标签，无引用 |
| 普通用户 | 40-55 | 有一定数据量，偶尔同步，基础标签 |
| 认真使用 | 60-75 | 数据完备，定期更新，标签规范 |
| 深度使用 | 75-90 | 高质量内容，全面覆盖，被同事引用 |
| 极少数标杆 | 90+ | 所有维度接近满分 |

## 后端 API

### GET /api/assets/score

**Response Schema：**

```python
class SubScoreDetail(BaseModel):
    key: str           # "quality_avg"
    label: str         # "ETL 质量均分"
    weight: float      # 0.4 (子指标在维度内的权重)
    score: int         # 34 (该子项得分，已按子权重折算)
    max_score: int     # 50 (该子项满分，= 维度满分100 × 子权重)
    value: str         # "0.68" (用户当前原始值，用于展示)
    criteria: list[str] # ["≥0.85 → 45-50分", "0.7-0.85 → 35-45分", ...]

class ScoreDimension(BaseModel):
    key: str                       # "quality"
    label: str                     # "内容质量"
    weight: float                  # 0.30 (维度在总分中的权重)
    score: int                     # 62 (维度总分 0-100)
    detail: str                    # "质量均分 0.68 · 高质量占比 42% · 字段完整 85%"
    sub_scores: list[SubScoreDetail]  # 子指标明细
    action: ScoreAction | None     # score < 70 时的优化建议

class ScoreAction(BaseModel):
    label: str         # "去导入数据"
    route: str         # "/data-import"

class AssetScoreResponse(BaseModel):
    total_score: int                    # 加权总分
    level: str                          # "良好"
    dimensions: list[ScoreDimension]    # 5 个维度
```

## 后端实现要点

### 参评内容过滤

```python
# 参评的 Communication 过滤条件（已确认：comm_type 枚举值为 'meeting', 'chat', 'recording'）
MEETING_COMM_TYPES = ["meeting", "recording"]  # 排除 'chat'
scored_comm_filter = Communication.comm_type.in_(MEETING_COMM_TYPES)
```

### 阶梯插值函数

```python
def _tier_score(value: float, tiers: list[tuple[float, float, int, int]]) -> int:
    """
    阶梯内线性插值。
    tiers: [(lower_bound, upper_bound, min_score, max_score), ...] 从高到低排列。
    upper_bound 为该区间的上限（最高档的 upper_bound 为理论最大值，如 1.0 或 100.0）。
    """
    for lower, upper, min_s, max_s in tiers:
        if value >= lower:
            if upper <= lower:
                return max_s
            ratio = min((value - lower) / (upper - lower), 1.0)
            return min_s + int(ratio * (max_s - min_s))
    return 0
```

**示例调用：**
```python
# ETL 质量均分 (子权重40%, 满分40)
quality_avg_score = _tier_score(avg_quality, [
    (0.85, 1.0,  36, 40),
    (0.70, 0.85, 28, 36),
    (0.50, 0.70, 20, 28),
    (0.30, 0.50, 12, 20),
    (0.00, 0.30,  0, 12),
])
```

### 边界情况处理

- **零数据/新用户**：当任何比率计算的分母为 0 时，该子指标得分为 0（与现有 `_ratio_score` 行为一致）
- **无参评内容**：所有维度得分为 0，总分为 0，等级"待提升"
- **仅有一种内容类型**：类型覆盖子项得 40 分，其他子项正常计算
- **无原创内容**：数据影响力维度的"被引内容覆盖率"分母为 0，该子项得 0 分

### 性能考虑

- 所有子指标查询在单个 API 调用中完成
- 可复用现有的 `_count()` 和 `_log_score()` 辅助函数
- 新增 `_tier_score()` 用于阶梯插值
- 参评内容筛选条件需在多个子查询间共享，避免重复查询

## 删除/废弃的内容

- 移除维度：`activity`（活跃度）、`sources`（数据源激活）、`rules`（数据治理规则）
- 移除维度：`knowledge`（知识图谱）— 被 `impact`（数据影响力）替代
