# 流光数据平台 - 飞书应用部署方案

## 目标

将流光数据平台发布为飞书网页应用（H5），支持多套飞书租户环境共用一套后端服务，数据默认共享、支持按租户筛选。

## 总体决策

| 项目 | 决定 |
|------|------|
| 应用形态 | 飞书网页应用（H5），在飞书工作台内打开 |
| 使用范围 | 企业内部，多套飞书租户 |
| 部署架构 | 一套后端 + 一套数据库，多套飞书 app 配置 |
| 数据策略 | 默认共享数据池，支持按租户筛选 |
| 租户识别 | URL 参数自动识别（?tenant=xxx），用户无感知 |
| 服务器 | 云服务器（公网 IP + 域名） |

---

## 一、数据模型变更

### 1.1 新增 `feishu_tenant` 表

| 字段 | 类型 | 说明 |
|------|------|------|
| id | Integer, PK | 自增主键 |
| code | String, Unique | 租户标识，用于 URL 参数（如 "a", "b"） |
| name | String | 显示名称（如 "总部环境", "分公司环境"） |
| app_id | String | 飞书自建应用 app_id |
| app_secret | String | 飞书自建应用 app_secret（加密存储） |
| redirect_uri | String | OAuth 回调地址 |
| webhook_url | String, 可选 | 该环境的告警 webhook 地址 |
| base_domain | String, 可选 | 飞书租户域名（如 xxx.feishu.cn） |
| is_active | Boolean | 是否启用，默认 true |
| created_at | DateTime | 创建时间 |
| updated_at | DateTime | 更新时间 |

### 1.2 `User` 表新增字段

| 字段 | 类型 | 说明 |
|------|------|------|
| tenant_id | Integer, FK | 外键关联 feishu_tenant.id，记录用户来自哪个租户 |

### 1.3 数据筛选逻辑

现有的 Document / Meeting / ChatMessage 等内容表不需要改动：

- 数据已有 `owner_id` 关联到 User
- User 有了 `tenant_id` 后，按租户筛选 = 按 `owner.tenant_id` 过滤
- 在列表页加一个"环境筛选"下拉框，调 API 时传 `tenant_id` 参数
- 默认行为：不筛选 = 看到所有数据（共享池）；选择某个环境 = 只看该租户的数据

## 二、认证流程改造

### 2.1 现有流程

```
用户打开网页
  -> 前端用固定的 app_id 跳飞书 OAuth
  -> 拿到授权码
  -> 后端用固定的 app_secret 换 token
  -> 创建/更新用户
  -> 签发 JWT
```

### 2.2 改造后的流程

```
用户从飞书工作台点击应用（URL 带 ?tenant=a）
  -> 前端读取 tenant 参数
  -> 调后端 GET /api/tenant/{code}/config 获取该租户的 app_id 和 redirect_uri
  -> 前端用该 app_id + redirect_uri 跳飞书 OAuth
  -> 飞书回调，前端拿到授权码
  -> 前端 POST /api/auth/feishu/callback，body 带上 code + tenant_code
  -> 后端根据 tenant_code 查 feishu_tenant 表，拿到 app_id + app_secret
  -> 用该凭证调飞书 API 换取用户信息
  -> Upsert 用户（绑定 tenant_id）
  -> 签发 JWT（payload 里带 tenant_id）
```

### 2.3 新增/修改的 API

| 接口 | 方法 | 说明 |
|------|------|------|
| `GET /api/tenant/{code}/config` | GET | 公开接口，返回 app_id 和 redirect_uri（不返回 secret） |
| `POST /api/auth/feishu/callback` | POST | 改造：body 新增 tenant_code 字段 |
| `CRUD /api/admin/tenants` | ALL | 管理后台：租户配置的增删改查（仅 admin 可操作） |

### 2.4 前端改动

- **登录页**：不再从 `import.meta.env` 读固定的 `VITE_FEISHU_APP_ID`，改为从 URL 取 `tenant` 参数，调接口拿 `app_id`
- **localStorage**：登录成功后存储 `tenant_code`，后续请求可带上
- **兼容处理**：如果 URL 没有 `tenant` 参数（比如直接浏览器访问），显示一个租户选择页面或使用默认租户

### 2.5 FeishuClient 改造

现在的 `feishu_client` 是全局单例，用固定 `settings` 里的凭证。改造为方法级传参：

```python
# 之前
app_id = settings.feishu_app_id
app_secret = settings.feishu_app_secret

# 之后
tenant = await get_tenant_by_code(db, tenant_code)
app_id = tenant.app_id
app_secret = tenant.app_secret
```

核心变化：feishu_client 的方法增加 app_id + app_secret 参数，不再读全局配置。

## 三、飞书应用配置（每个租户都要做）

每个飞书租户环境需要在飞书管理后台完成以下操作：

1. 进入飞书开放平台 -> 创建自建应用
2. 记录 app_id 和 app_secret
3. 添加应用能力 -> 选择"网页应用"
4. 配置网页应用地址：`https://your-domain.com/?tenant={code}`
5. 配置 OAuth 重定向 URL：`https://your-domain.com/login?tenant={code}`
6. 开通所需权限：
   - 获取用户基本信息
   - 获取用户邮箱
   - 其他业务所需权限（日历、多维表格等）
7. 发布应用（企业内部发布，无需飞书审核）
8. 在流光平台管理后台填入该租户的 app_id、app_secret 等配置

## 四、部署架构

### 4.1 服务器要求

- 云服务器（阿里云/腾讯云等），建议 2核4G 起步
- 公网 IP + 域名（需备案）
- SSL 证书（飞书要求 HTTPS）

### 4.2 Docker Compose 部署

沿用现有的三容器架构，无需改变：

```
┌──────────────────────────────────────────┐
│               云服务器                     │
│                                           │
│  ┌──────────┐  ┌──────────┐  ┌─────────┐ │
│  │  Nginx   │  │ Backend  │  │ Postgres │ │
│  │ (前端+   │──│ (FastAPI)│──│(pgvector)│ │
│  │ 反向代理) │  │  :8000   │  │  :5432  │ │
│  │ :80/443  │  └──────────┘  └─────────┘ │
│  └──────────┘                             │
└──────────────────────────────────────────┘
         ↑
    飞书环境A的用户 (?tenant=a)
    飞书环境B的用户 (?tenant=b)
```

### 4.3 Nginx 配置要点

- 强制 HTTPS（飞书网页应用要求）
- 前端路由 fallback 到 index.html（SPA history 模式）
- `/api` 反向代理到后端 :8000

### 4.4 环境变量变更

由于飞书凭证改为数据库存储，`.env` 中的以下字段可保留作为默认/回退配置，但不再是唯一来源：

```env
# 保留为默认租户的回退配置
FEISHU_APP_ID=xxx
FEISHU_APP_SECRET=xxx
```

前端的 `VITE_FEISHU_APP_ID` 和 `VITE_FEISHU_REDIRECT_URI` 不再需要构建时注入，改为运行时从 API 获取。
